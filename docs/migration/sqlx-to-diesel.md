# Migration Plan: SQLx to Diesel

## Why?
The project currently uses `sqlx` and `sqlx-data` for database interactions. While `sqlx` provides compile-time query checking, `diesel` offers a robust ORM with a powerful query builder DSL, type safety, and schema management. Moving to `diesel` (specifically `diesel-async`) can improve maintainability, reduce boilerplate SQL strings, and leverage Diesel's ecosystem for migrations and schema management.

## Actual
- **Dependencies**: `sqlx`, `sqlx-data`, `sqlx-utils`.
- **Schema**: Implicitly defined by migrations and `sqlx` macros.
- **Models**: `backend-model` structs derive `FromRow`.
- **Repositories**: `backend-repository` uses `#[repo]` and `#[dml]` macros with external `.sql` files.
- **Migrations**: `backend-migrate` uses `sqlx::migrate!` with flat `.sql` files.
- **Pool**: `backend-server` uses `sqlx::PgPool`.

## Constraints
- **Async**: The application is async (Axum), so we must use `diesel-async` and an async connection pool (`deadpool-diesel` or `bb8-diesel`).
- **Postgres**: The database is PostgreSQL.
- **Existing Data**: Migrations must preserve existing schema structure.
- **Incremental Migration**: It's difficult to mix `sqlx` and `diesel` in the same transaction, but we can migrate crate by crate or module by module if we maintain two pools temporarily. However, a clean cutover is preferred for simplicity if feasible.

## Findings
- **Dependencies**:
    - `diesel` (core ORM)
    - `diesel-async` (async support)
    - `deadpool-diesel` (async pooling)
    - `diesel_migrations` (migration management)
- **Schema**: Diesel requires a `schema.rs` file generated by `diesel print-schema`. This should likely reside in `backend-model` or a new `backend-schema` crate to be shared.
- **Models**: `FromRow` structs in `backend-model` need to derive `Queryable`, `Selectable`, `Identifiable`, `Insertable`, `AsChangeset`.
- **Repositories**: `backend-repository` traits can remain, but implementations in `src/pg/*.rs` must be rewritten to use Diesel DSL.
- **Migrations**: Existing `.sql` migrations in `backend-migrate/migrations` are flat files. Diesel expects `up.sql` and `down.sql` in versioned directories.

## How to?

### 1. Dependency Changes

**Root `Cargo.toml`**:
```toml
[workspace.dependencies]
diesel = { version = "2.2", features = ["postgres", "chrono", "serde_json", "uuid", "r2d2"] }
diesel-async = { version = "0.5", features = ["postgres", "deadpool"] }
diesel_migrations = "2.2"
deadpool-diesel = "0.6" # or compatible version
# Remove: sqlx, sqlx-data, sqlx-utils
```

**Crate `Cargo.toml`s**:
- `backend-core`: Remove `sqlx`. Add `diesel` (for error types).
- `backend-model`: Add `diesel`. Remove `sqlx`.
- `backend-repository`: Add `diesel`, `diesel-async`. Remove `sqlx`, `sqlx-data`.
- `backend-migrate`: Add `diesel`, `diesel_migrations`, `diesel-async`. Remove `sqlx`.
- `backend-server`: Add `diesel`, `diesel-async`, `deadpool-diesel`. Remove `sqlx`.

### 2. Schema Management
- Install `diesel_cli`: `cargo install diesel_cli --no-default-features --features postgres`
- Run `diesel setup` to generate `diesel.toml` and `migrations/`.
- Configure `diesel.toml` to output `schema.rs` to `app/crates/backend-model/src/schema.rs` (or similar).
- **Action**: Create a `schema.rs` in `backend-model` that reflects the current database state.

### 3. Migration Porting
- Move existing SQL files from `app/crates/backend-migrate/migrations/*.sql` to Diesel format:
    - Create directories: `migrations/YYYYMMDDHHMMSS_name/`
    - Rename existing `.sql` to `up.sql` inside the directory.
    - Create empty `down.sql` (or implement revert logic).
- Update `backend-migrate/src/migrate.rs`:
    - Use `diesel_migrations::embed_migrations!`.
    - Implement a runner that uses `diesel-async` connection or a sync connection (Diesel migrations are typically sync, but `diesel-async` has adapters). *Note: `diesel_migrations` works with `diesel::PgConnection` (sync). It's often easier to run migrations synchronously at startup using a temporary sync connection.*

### 4. Model Updates (`backend-model`)
- Update structs in `src/db.rs`:
    - Add `#[derive(Queryable, Selectable, Identifiable, Debug, Clone)]`.
    - Add `#[diesel(table_name = ...)]`.
    - Ensure field types match Diesel mappings (e.g., `chrono::DateTime<Utc>`, `serde_json::Value`).
- Create separate `Insertable` structs if needed (or derive on existing if they match 1:1).

### 5. Repository Rewrite (`backend-repository`)
- Refactor `PgKycRepo` (and others) to use `deadpool_diesel::postgres::Pool`.
- Rewrite methods:
    - **Select**: `users::table.filter(users::id.eq(user_id)).first::<User>(conn).await`
    - **Insert**: `diesel::insert_into(users::table).values(&new_user).get_result(conn).await`
    - **Update**: `diesel::update(users::table.find(id)).set(&changes).execute(conn).await`
    - **Complex Queries**: Use `diesel::sql_query("...").bind::<Type, _>(value).load(conn).await` if DSL is too complex.
- Remove `#[repo]` and `#[dml]` attributes.
- Remove `queries/` directory after porting.

### 6. State & Pool Management (`backend-server`)
- Update `AppState`:
    - Replace `pool: PgPool` with `pool: deadpool_diesel::postgres::Pool`.
- Update `main.rs` / startup logic:
    - Initialize `deadpool_diesel::postgres::Pool`.
    - Run migrations (sync) before starting the async runtime or using `tokio::task::spawn_blocking`.

### 7. Step-by-Step Execution Plan

1.  **Preparation**:
    -   [ ] Create a new branch `feature/migration-diesel`.
    -   [ ] Install `diesel_cli`.

2.  **Infrastructure**:
    -   [ ] Update `Cargo.toml` dependencies.
    -   [ ] Initialize Diesel: `diesel setup`.
    -   [ ] Port migrations to `migrations/` directory.
    -   [ ] Generate `schema.rs` (`diesel print-schema > app/crates/backend-model/src/schema.rs`).

3.  **Core & Model**:
    -   [ ] Update `backend-model` to use `diesel` traits and `schema.rs`.
    -   [ ] Fix `backend-core` if it has `sqlx` dependencies (e.g., errors).

4.  **Migration Runner**:
    -   [ ] Rewrite `backend-migrate` to use `diesel_migrations`.

5.  **Repositories (Iterative)**:
    -   [ ] Port `backend-repository/src/pg/user.rs`.
    -   [ ] Port `backend-repository/src/pg/kyc.rs`.
    -   [ ] Port `backend-repository/src/pg/device.rs`.
    -   [ ] Port `backend-repository/src/pg/approval.rs`.
    -   [ ] Port `backend-repository/src/pg/sms.rs`.

6.  **Server Integration**:
    -   [ ] Update `backend-server` state and startup to use `deadpool-diesel`.
    -   [ ] Fix compilation errors in handlers (if any leak `sqlx` types).

7.  **Cleanup**:
    -   [ ] Remove `sqlx` dependencies and `queries/` folder.
    -   [ ] Verify all tests pass.

## Conclusion
This migration moves the project to a strongly-typed, ORM-based approach with Diesel. While it requires significant refactoring of the repository layer, it eliminates raw SQL maintenance and leverages Diesel's robust ecosystem. The use of `diesel-async` ensures we maintain the async nature of the application.
